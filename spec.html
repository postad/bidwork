BidWork — multi-tenant auto-bidding for trade subcontractors

BidWork takes an incoming bid invitation (ITB/RFP), reads the drawings and specs, and auto-drafts a professional, branded proposal for the subcontractor — priced from their own charged prices. The contractor reviews, optionally edits, and sends in one click. Replies route straight back to their inbox. Alongside bidding, the platform builds the contractor's network by surfacing every relevant person found in the project documents and sending warm "say-hi" intros.

Brand  BidWork First tenant  The Shade Company (commercial window treatments, NYC metro) Predecessor  BidER MVP (single-company, retired) Direction  Greenfield rebuild

This document is the source of truth for the rebuild. The current repo (BidER) is a tiny single-company MVP whose UI/UX and data model are wrong for a multi-tenant product — start fresh, carry over only what's noted.

1 · What we're building

A multi-tenant SaaS that automates the most painful, repetitive part of running a specialty trade business: responding to bid invitations. The operator (admin) sources bid opportunities and distributes them to subscribing contractors, each receiving a finished draft bid built from their own pricing — effectively a lead-distribution + auto-bid marketplace, not just a self-serve tool.

Phase 1

Operator sources ITBs, uploads them, and the system assigns each to the relevant contractors with an auto-drafted bid ready to review.

Phase 2

Contractors upload their own bid documents directly; the same pipeline drafts the proposal.

2 · Who it's for

Local service contractors who bid frequently and price from repeatable units. We start with window treatments and expand across local trades. The architecture is vertical-agnostic: a per-vertical knowledgebase defines what "a unit" and "a price" mean for each trade (see §4 and UPGRADE-PLAN.md).

Window treatments · first Epoxy floors Garbage removal & cleaning HVAC …more local services
Two roles:
  • Admin / operator — sources bid-request documents and uploads each with a geo location as a radius (e.g. "100 mi from zip 10018"). Admin does not pick the trade — the system suggests it (see below).
  • Contractor (tenant user) — sets up their Pricing DNA, receives ready bids, reviews/edits/sends, grows their network.
Auto trade-relevance + geo dispatch. On upload, the system scores the document against every trade in the system and suggests bid / no-bid per trade — one bid can apply to several trades, while trades with no relevant scope score no-bid and drop out. Each relevant trade then dispatches a tailored, individually-priced proposal to every contractor in that trade within the geo radius.

3 · Core values — the things that must be excellent

Accuracy in bidding

Drafts must be trustworthy, and the system learns from every bid as the data grows (per-tenant).

One-click send

From "bid ready" to "delivered to the GC" with a preview and a single confirm.

Network building

Warm intro emails to GCs, architects, owners — get on more bid lists. Considered possibly the biggest value-add.

Ease of use

A contractor should never feel like they're operating software.

4 · The bid engine The core ★

This is the heart of the product: take a 100–200 page bid-request document (drawings + specs) and produce a finished, sendable proposal for each matched contractor, priced from their own Pricing DNA. Everything else (network, send flow) orbits this. Detailed design lives in UPGRADE-PLAN.md — summary below.

Pipeline · one document → relevant trades → one proposal per matched contractor
  1. 1Route & render — a deterministic router pre-classifies pages; each renders to an image with rotation auto-corrected and large drawing sheets tiled. Page-hash caching makes re-runs cheap; respects the 32 MB / 100-page API limits.
  2. 2Triage (Haiku) — cheaply identifies the pages that actually matter (schedules, plans, relevant CSI sections) so the expensive model only reads what counts.
  3. 3Score trades (no-bid gate) — evaluate the package against every vertical's signals and emit a bid / no-bid relevance score per trade. One document can be relevant to several trades; trades with no scope score no-bid and drop out. Admin sees these suggestions on upload. Reuses the MVP's yes-bid / no-bid signal logic (e.g. the 7 Brew drive-thru → no window scope pattern).
  4. 4Extract (Opus) — for each relevant trade, structured extraction into that vertical's schema: units, room types, area, attributes (motorization, blackout vs. solar…), timeline, contacts. Every field carries a confidence score + page citation, and the result is schema-validated. ~$1.50/project. Replaces the MVP's regex scanner.py.
  5. 5Flag gaps — measure the extracted scope against the vertical's required-evidence checklist + the document reference graph, and flag anything required-but-empty, low-confidence, conflicting, or pointing to a sheet that wasn't uploaded. Absence is only detectable against an expectation — see the card below.
  6. 6Price from Pricing DNA — apply the contractor's charged prices (per-unit / per-sqft / per-ton / hourly / mixed) to the extracted scope. Per-tenant — never another contractor's numbers.
  7. 7Draft the proposal — assemble a branded, client-ready document per contractor (one per relevant trade × matched contractor): scope, line-item pricing, exclusions, terms, timeline — using their logo + boilerplate. Pricing is grouped by location to match the drawings (see Proposal structure below).
Decided · proposal structure

Group the proposal by location, not by product. Plans are organized by floor → room/area/zone, so the proposal mirrors that — Floor 23 · Boardroom 2302 · 3 motorized rollers · $4,440 — instead of one product roll-up (14 motorized × $1,480). It proves we read the drawings and lets the GC verify line-by-line against their own set. The grouping dimension is vertical-configurable (floor/room for shades & cleaning, area/slab for epoxy, system/zone for HVAC) — one pattern, every trade.

  • Pricing DNA is unchanged — each location line still ties to a charged-price item (per-unit/sqft/LF). We only re-group the same line items; learning-loop diffs still work per line.
  • "By product" toggle — a one-click roll-up by product/system for the GC's value-engineering view. Location is the default; product is the summary.
  • Per-line "what we read" note — a tap-to-open, plain-language tooltip on each location line (source sheet, assumptions, any open flag). It's a contractor-facing sanity check during review — confirm our read matches reality — and does not print on the GC proposal. The separate "Notes to GC" block (§6) is what prints. Prototyped in bid.html.

Why not regex: text-only scraping can't count units, garbles drawing text, and reads only a handful of title blocks. Vision reading the rendered pages is what makes the scope trustworthy.

Decided · single-pass multi-trade scan

One upload is read once and scored against every trade in the system — not a per-trade upload. The expensive 100–200 page read (render + route + relevance scan) happens a single time and emits a bid/no-bid score per trade in one pass; the costly Opus extraction then runs only for the trades that scored bid. A package relevant to 1 of 6 trades costs ≈ one scan + one extraction, not six full runs.

  • Prompt caching caches the document so any per-trade calls pay the document tokens ~once (~10% on cache reads); combined with the page-hash render cache, re-runs and multi-trade scoring are cheap.
  • The relevance gate is semantic, not keyword — validated on a real epoxy package where all 12 “epoxy” mentions were anchors/adhesive, not flooring (a keyword router would have mis-bid it). Borderline scores are admin-confirmed, never silently dropped.
  • Validated across three live verticals (window treatments priced to the penny; cleaning; epoxy). See spike/FINDINGS.md and BUILD-PLAN.md.
How gaps are detected — including the "unknown unknowns"

You can't detect an absence in a vacuum. A gap is only knowable relative to an expectation, so the system compares what extraction returned against the vertical's required-evidence checklist + the document reference graph — not against "what the model happened to notice." Five distinct kinds of gap, five detectors:

1 · Required-but-emptyThe vertical config declares the fields needed to price the job. Any required slot returned null or below a confidence threshold → flagged. The checklist is what makes the gap knowable.
2 · Low-confidence / conflictingFound but uncertain, or two sources disagree (schedule says 5% openness, spec says 3%). Cross-source check flags the conflict with both citations.
3 · Internal inconsistencyElevations imply ~40 windows; the schedule lists 32. Quantity reconciliation across drawings vs. schedule flags the mismatch.
4 · Missing documentThe true unknown unknown. Two mechanisms: an expected-document-set check (triage classifies which doc types / CSI sections are present; if the vertical expects a window schedule and none was classified → flag), and reference-chasing (a sheet cites "see A-601" or "spec 12 24 00"; if it isn't in the package → flag).
5 · No-bid / out-of-scopeDetect whether it's even biddable (e.g. drive-thru-only projects have no window scope).

The model must be allowed to say "not found" — structured output with explicit nulls + reasons, so it never hallucinates a value to fill the schema. A confident-but-wrong number is more dangerous than a flagged blank.

Honest residual limit: info that's genuinely absent, unreferenced, and off-checklist can't be guaranteed-caught. Mitigations: every assumption is surfaced explicitly in the proposal ("Assumes 5% openness — confirm") never silently; conservative defaults are always flagged; human-in-the-loop review is the backstop; and the learning loop lowers a field's prior confidence when contractors keep correcting it.

Who sees which gaps

Two audiences, two tiers. Admin owns the integrity of the request; the contractor owns the integrity of their proposal against that request.

Admin · at upload

Adds the trade + the documents. Sees document-level gaps: references to sheets/specs that weren't uploaded (e.g. "cites A-601 — not in package") and other completeness gaps within the bid request itself. These are the same for everyone the bid dispatches to, so admin resolves/chases them before dispatch. Gating is severity-based: a critical gap (pricing impossible) blocks dispatch; everything else dispatches with a visible caveat. Which evidence is bid-blocking is declared per trade in the vertical config.

Contractor · in bid review

Sees the gap between the bid request and their own proposal — the scope/pricing items to confirm to finalize their bid ("We read 32 windows — confirm?", "Assumes 5% openness — change?"), as plain-language flags inside the bid (§6). Never the raw scan.

Raw extraction output (confidence + citations) is an internal/QA artifact — neither party reads it as raw data.

Pricing DNA

The contractor's pricing model — charged prices only, learned from their own past bids (onboarding calls it "your pricing DNA"). It's what makes every drafted bid theirs. Stored and trained per-tenant; improves with every edit. Never cost or margin (see §10).

One engine, config per vertical

A single config-driven engine — not a scanner per trade. Each vertical is one JSON knowledgebase (units, products, attributes, pricing model, scan keywords, CSI map, exclusions, terms, and a requiredEvidence completeness checklist + expected-document set that the gap-detection step measures against) + an agent prompt. Adding a trade = adding a config, not writing code.

Accuracy & the learning loop

Nothing auto-sends. Extraction is confidence-gated with human-in-the-loop review. The system then learns from the gap between the draft and the contractor's edits — field-level diffs (quantity / price / line-item / wording) feed back into their Pricing DNA. This forces bids to be stored as structured data, not DOCX blobs, so every edit is diffable; the PDF/DOCX is an output, not the source of truth.

Why it's the moat: accuracy compounds. Each reviewed/edited bid sharpens that contractor's Pricing DNA, and cross-tenant pooling of document-understanding patterns (never pricing) makes extraction smarter for everyone. The engine that reads a 200-page set correctly and prices it in the contractor's own voice is the hard, defensible part.

5 · End-to-end flow

  1. 1System generates a base questionnaire per vertical.
  2. 2Contractor answers it about their pricing → saved as their Pricing DNA (charged prices only — see constraints).
  3. 3Admin uploads a bid document (ITB/RFP) with a geo radius (e.g. 100 mi from 10018). System scores it against every trade and suggests bid/no-bid per trade. (Phase 2: contractors upload directly.)
  4. 4For each relevant trade, system dispatches to matching contractors within the geo radius (trade + radius + active ITB).
  5. 5The bid engine (§4) reads the document and auto-generates a branded proposal per contractor from their Pricing DNA + the extracted scope.
  6. 6Contractor gets an email: "You have a new bid ready."
  7. 7They click through → view the bid → Approve or Edit.
  8. 8Pricing is shown grouped by location (floor/room/zone) with a "By product" roll-up toggle; each location line has a tap-to-open "what we read" tooltip. Edit mode allows: discount (% or $ + label), add line items, add a "Notes to GC" block. Pricing recomputes live.
  9. 9On Approve → a final PDF preview + editable email appears (warm intro, no numbers; "send me a copy" on by default).
  10. 10Send → proposal PDF emails to the bid contact, reply-to = the contractor's own inbox.
  11. 11Network: everyone found in the project docs is surfaced; the contractor sends "say-hi" intros and those people join their network.

6 · The system, screen by screen

These reflect decisions made while building the prototype. Each links to its mockup.

Onboarding  open ›

Captures company identity (logo, description, address) and the base pricing model. Read-only on the proposal boilerplate — a small pointer tells the user they can edit terms/exclusions/disclaimers later in Settings. Upload-first hybrid: prefer extracting charged prices from past bids over asking.

Settings  open ›

The single editable home for proposal boilerplate: payment terms, warranty, quote-valid window, an editable exclusions list, and a disclaimer. Logo upload lives here too. No cost/margin fields anywhere.

Dashboard  open ›

Linear-style list of bids — New bids / In progress — plus Network and Contacts entry points. The contractor's home base.

Bid (personalization)  open ›

Where the product proves it read the drawings carefully. Pricing is grouped by location (floor → room/zone) to match the drawings, with a "By product" toggle for a value-engineering roll-up, and a tap-to-open "what we read" tooltip per line (source / assumptions / flags) — a contractor-only sanity check that doesn't print to the GC. View mode shows the drafted proposal; Edit mode turns room labels, line items, quantities, and unit prices into editable fields with a live-recomputing total, lets the user add line items, add a discount (% or $ toggle + custom label), and add one "Notes to GC" free-text block. The right panel shows a charged-price breakdown (subtotal / delivery / discount / total) — never cost or margin. See §4 "Proposal structure".

Send → preview → PDF

"Approve & send" opens a modal with a final PDF preview of the actual proposal (with logo), a Download PDF option (print isolation), and an editable email composer. The email is a warm intro with no numbers — the PDF holds the detail. "Send me a copy" is checked by default. Reply-to is the contractor's address.

Sent (receipt)  open ›

Delivery receipt confirming who it went to, reply-to, attachment, and time. Reassures the contractor that replies come straight to them and the bid is tracked.

7 · Network & "say-hi"  open ›

Reframed from a passive "Contacts" address book into an outbound relationship engine with a LinkedIn-style feel. The system surfaces highly relevant people matched to the contractor's area (NYC metro) and vertical, sourced from the project PDFs — not just GCs, but architects, owners, and designers.

Email is the unit — no email, no contact.

A contact is only actionable if extraction found an email — saying hi is an email, so a name with no address is dead weight. The email shows on the row as proof it's reachable. People found without one are excluded, surfaced only as an honest footnote ("2 more found, no email — not added").

Warm · you bid it

A GC whose project the contractor already bid — lead with that as the match reason.

No-scope · no-bid

The trade scored no-bid on this project, but the person is in-area — still a warm reason to say hi. A no-bid is never a dead end.

Cold / new

A strong new match on territory + trade + active ITBs.

It's a "say-hi", not a chase. Each hello sends on a gentle timer — never a blast — and the moment it goes out, that person lands in the contractor's network. No follow-up sequences. Tone adapts to scope:
  • Bid: "…we recently submitted a bid on [project], and we'd love to be at your service."
  • No-scope: "I noticed your project at [project] — not sure window treatments are in scope from the documents, but we work right in your area and would love to be on your radar."
Editable before send; replies route to the contractor's inbox.

Decoupled from the bid: per-trade scoring is effectively three-way — BID (priced proposal + contacts) · NO-BID but in-area (no proposal, contacts still flow as say-hi) · truly irrelevant/out-of-range (nothing). UI is a list: Contact / Type / Found in / Action, with type filters and segments (All / Suggested / In your network).

8 · Reply tracking — WhatsApp-style, kept simple for v1

Replies live in the contractor's own inbox (v1 — no in-app threading). The platform shows lightweight status so they know where each outreach stands:

Queued → ✓ Delivered → ✓✓ Read → Replied
  • Single ✓ (Delivered) — from the Mailgun delivery webhook. Reliable.
  • Double ✓✓ (Read) — open-tracking pixel. Imperfect (some clients block images, so a read may only ever show as Delivered) — acceptable for v1.
  • Replied — flips when the contractor gets a reply; for v1 the reply itself is read/answered in their normal inbox.

Future option (not v1): a per-thread reply alias that forwards to the inbox so replies can be auto-detected and a light in-app thread shown. Deferred to keep it simple.

9 · Tech stack

Supabase — all-in
  • Postgres as the database.
  • Storage for uploaded ITB/RFP PDFs, per-tenant logos, and generated bid files.
  • Auth — to avoid hand-building password reset, email verification, OAuth (Google/Microsoft), sessions. Retires the MVP's NextAuth.
  • Row-Level Security (RLS) for tenant isolation (workspace-scoped rows).
Mailgun — email (single ESP, two streams)
  • Bids + replies on a bid subdomain; intro/outreach on a separate subdomain (reputation isolation — outreach complaints can never harm bid deliverability).
  • Chosen for mature inbound parsing (the reply round-trip) and delivery webhooks (powers the ✓ states). Volume < 200 intro emails/mo.
  • Per-tenant sending identity via OAuth mailbox-connect ("Connect Google / Microsoft") — a revocable token, never a stored password. DNS subdomain delegation (SPF/DKIM) is the send-only, no-OAuth fallback. (Login to the dashboard is separate from connecting a mailbox to send.)
  • Supersedes the earlier Resend plan.

Decided: Mailgun handles both streams (separate subdomains). Revisit a dedicated cold-outreach tool only if outreach volume or deliverability ever demands it.

Document extraction — two-pass vision pipeline

Haiku triage → Opus structured extraction, ~$1.50/project. Replaces the MVP's scanner.py. Reads drawings/specs for window count, room types, scope, motorization, blackout vs. solar, timeline, and contact info. Needs ANTHROPIC_API_KEY.

Frontend — Next.js 14 + Tailwind

Kept from the MVP; rebuild the UI under the BidWork brand. Design tokens are the bw- palette used across these mockups (green #14A800, Inter + JetBrains Mono).

Hosting — Vercel
  • The Next.js app deploys to Vercel — native Next.js support, preview deploys per PR, edge/serverless functions for the API routes (extraction kickoff, Mailgun webhooks).
  • Routing: the root domain (/) serves the marketing homepage; the product app lives under /app (dashboard, bid review, Network, settings, operator console). One Next.js project, split by route group.
Decided · data layer

All-in on Supabase with full Auth. Supabase Auth owns the whole identity surface — email/password, email verification, password reset, OAuth (Google/Microsoft), and sessions (retires the MVP's NextAuth). supabase-js is the data layer, since it passes the per-request auth context that RLS policies read. Prisma is dropped (it can't pass that context natively). RLS enforces tenant isolation on every workspace-scoped row.

Data model

Not a complicated schema — ~13 tables in five groups. The rule of thumb: relational columns for anything we query, filter, or learn from (prices, line items, edits, statuses); JSONB for shape-varying payloads (raw extraction, per-vertical config, document metadata). Everything workspace-scoped carries a workspace_id that RLS keys on.

Tenancy & identity
  • workspaces — one per contractor tenant (the RLS boundary).
  • profiles — users, linked to auth.users; role (admin / contractor), branding (logo, address, description), reply-to email, connected-mailbox token ref.
Trades & pricing
  • trades — global vertical catalog (window treatments, epoxy, HVAC…); holds the VerticalConfig JSONB (schema, requiredEvidence checklist, no-bid rules).
  • workspace_trades — which trades a tenant serves + their geo (center zip, lat/lng, radius_mi).
  • pricing_items — the Pricing DNA: sell price only per unit / sqft / linear ft / adder. Never cost or margin.
Requests & extraction
  • bid_requests — an admin upload: geo radius, trade-relevance scores, document-level gaps, dispatch status.
  • documents — each uploaded file in Storage + page metadata, classification, page-hash cache key (JSONB).
  • extractions — per request × trade: the raw structured extract (JSONB) with per-field confidence + page citations. Internal/QA artifact.
Bids & learning
  • bids — one per matched contractor; status (draft / approved / sent), boilerplate snapshot frozen at send (immutable record).
  • bid_line_items — relational rows (qty, unit, sell price) so every field is diffable.
  • bid_edits — the learning loop: field-level diffs between AI draft and contractor edit, categorized quantity / price / line-item / wording. Feeds that tenant's Pricing DNA only.
Network & email
  • contacts — GCs / architects per tenant (built from intro outreach).
  • emails — every outbound (bid or intro): stream, status timeline (queued → delivered ✓ → read ✓✓ → replied), Mailgun message id. Powers the Network status key.
Billing & plans · schema now, enforce later
  • plans — global catalog: monthly price, included bids / month, Stripe price id.
  • subscriptions — per workspace: Stripe customer/subscription ids, plan, status (trialing / active / past_due / canceled), trial_bids_used (limit 3), current period start/end.
  • bid_usage_events — a ledger row per billable bid (workspace, bid, period, counted_at) — meters consumption against the monthly cap; one row per bid avoids double counting.

Three correctness ties: (1) learning loop works because line items are relational rows + bid_edits captures every diff; (2) pricing privacy holds because pricing_items stores sell price only and RLS scopes it to one workspace; (3) immutable sent bids — the boilerplate snapshot on bids means a sent proposal never changes even if base settings later do.

10 · Plans & billing — Stripe · planned, not immediate

New contractors onboard and get a free trial of 3 bids (count-based, not a clock). After that they subscribe to a monthly package, and each package includes a fixed number of bids per month — the system caps bids per month per package. Stripe runs the subscriptions. We are building the data model and scope to support this now; enforcement lands in a later phase.

What counts as a “bid”

A draft-ready, dispatched bid to a contractor (a bids row that reaches “ready to review”). No-bid trades, failed extractions, and never-dispatched drafts don’t count. Each billable bid writes one bid_usage_events row. (Exact trigger — draft-ready vs. sent — is an open decision.)

Where the cap is enforced

At admin dispatch (Phase 1). Each target contractor is checked against their remaining allowance (trial credits, else plan period allowance); over-limit contractors are skipped with a visible reason — the same pattern the Review & dispatch screen already uses for “incomplete Pricing DNA”. v1 = hard cap + upgrade prompt; overage billing deferred.

Stripe integration

Checkout to subscribe, Customer Portal for self-serve management, and webhooks (customer.subscription.*, invoice.paid / payment_failed) to sync subscriptions.status + period dates. plans map to Stripe Prices; the monthly allowance resets on period rollover. Schema lands in foundation so there is no painful migration later — see BUILD-PLAN.md §5.

11 · Hard constraints — non-negotiable

Never ask for cost, margin, markup, or supplier discounts.

The base pricing model stores only what the contractor charges the customer (sell price per unit / sqft / linear ft / adder). A sendable bid only needs sell prices. Prefer extracting charged prices from past bids over asking.

Pricing is private — and the promise holds architecturally.

Show a visible privacy statement at onboarding ("encrypted, never shared with GCs, never shown to other contractors, never used to build anyone else's bids"). RLS enforces tenant isolation. The pricing learning loop is per-tenant — a contractor's prices/edits train only their own model. Cross-tenant pooling is allowed only for document-understanding/extraction patterns, never pricing.

BidWork stays out of the conversation.

Every outbound (bid or intro) sets reply-to = the contractor's own email. Replies go straight to them. The platform tracks status but does not insert itself into the relationship (v1).

Never send without review.

A contractor always sees the total and the final PDF preview before anything goes out. Defaults: 50% deposit / 50% on completion; 2-year labor + manufacturer product warranty; always include an exclusions section.

12 · Decisions

Decided
  • Greenfield rebuild. New project, not an evolution of the MVP. The in-repo UPGRADE-PLAN.md / WEBSITE-PLAN.md predate this and assumed evolving the MVP — treat them as engine reference only.
  • Admin-only sourcing (Phase 1). The operator uploads and dispatches bid requests; direct contractor upload is deferred to a later phase.
  • Reply tracking = inbox-only + ✓/✓✓ status. Replies land in the contractor's own inbox; the app shows delivered (✓) / read (✓✓) / replied status. Implemented and documented in the Network screen; the per-thread relay alias stays deferred.
  • Structured bid storage. Bids are structured data (line items as relational rows + bid_edits diffs); DOCX/PDF is output only. Schema is in §9 Data model.
  • Single-pass multi-trade scan. One upload is read once and scored against every trade; Opus extraction runs only for bid trades; prompt caching + page-hash cache keep it cheap. Relevance gate is semantic and admin-confirmed (§4).
  • Billing = Stripe, trial + capped monthly packages. 3-bid free trial, then a monthly package with a fixed bid cap; enforced at dispatch. Schema lands in foundation; enforcement is a later phase (§10).
  • Build approach = walking-skeleton vertical slice. One thin end-to-end slice across both roles first (window-treatments leads the full pipe; the scan is multi-trade from day one), then widen. See BUILD-PLAN.md.
Still open
  • Phase 2 timing: when to open direct contractor uploads.
  • First-slice trade scope: window-treatments leads the full pipe (default) vs. all three trades end-to-end at once.
  • Bid metering trigger: a billable bid counts at draft-ready (proposed) vs. only on send.
  • Overage policy: hard cap + upgrade prompt (v1) vs. metered overage billing (later).
